# 实验四：Hadoop 与 Spring Boot 融合实验 — 架构与答辩说明

> 项目：GengDirection（梗社区）｜后端 Spring Boot 4 + Java 21 ｜前端 Vue3
> Hadoop 集群：`root@218.194.47.8:5222`，Hadoop 3.4.1 单机伪分布式

---

## 1. 一句话讲清这个实验做了什么

> 用户在梗社区里的**浏览/收藏/评论/发帖**行为，被 Spring Boot 埋点采集，
> 批量写进 **HDFS**；在服务器上用 **MapReduce** 算出「热门帖子 Top-N」和
> 「行为类型分布」；结果再通过 REST 接口被前端 **ECharts** 画成柱状图和饼图。
> ——这就是企业里典型的「业务系统 + 大数据平台」融合架构。

## 2. 架构与数据流

```
 ┌─────────── 数据采集层 (Spring Boot) ───────────┐
 用户操作 ─► BehaviorInterceptor 自动埋点 ┐
 前端 ───► POST /api/behavior/track ──────┤
                                          ▼
                                  BehaviorCollector  (内存队列, 30s/200条批量)
                                          │ Hadoop FileSystem Java API
                                          ▼
 ┌──────── 分布式存储 (HDFS) ────────┐
 hdfs://218.194.47.8:10035
 /geng/behavior/raw/yyyyMMdd/events-*.log   每行 TSV: userId⇥action⇥postId⇥ts
                                          │  hadoop jar geng-mr-1.0.jar
                                          ▼
 ┌──────── 批量处理 (MapReduce / YARN) ────────┐
 HotPostJob   : Mapper 按行为加权 → Reducer 求和 → Top-N
 ActionDistJob: Mapper (action,1) → Reducer 求和 (WordCount 变体)
                                          ▼
 /geng/behavior/out/hotpost/part-r-00000      postId⇥heat
 /geng/behavior/out/action-dist/part-r-00000  action⇥count
                                          │  HdfsService.readLines + 关联 MySQL 标题
                                          ▼
 ┌──────── 展示层 ────────┐
 GET /api/analytics/hot-posts ─► ECharts 横向柱状图
 GET /api/analytics/action-distribution ─► ECharts 饼图
```

## 3. 实验大纲要求 → 实现对照

| 大纲要求 | 本项目实现 |
|---|---|
| Spring Boot REST 接收业务数据存 HDFS | `BehaviorController` + `BehaviorCollector` + `HdfsService`（Hadoop Java API） |
| MapReduce 标准 Mapper/Reducer | `geng-mr` 模块：`HotPostJob`、`ActionDistJob`，均用 `ToolRunner` |
| Spring Boot 操作 HDFS（HadoopTemplate 或 Java API） | 用 **Hadoop Java API**（`FileSystem`），Spring Data Hadoop 已停更不兼容 SB4 |
| 可视化处理结果（ECharts/表格） | 前端 `DashboardView.vue` 用 ECharts 画柱状图+饼图 |
| 前后端 RESTful | 统一 `Result<T>` 包装，资源路径语义化 |
| 用户行为分析案例 | 热门帖子 Top-N + 行为类型分布 |

## 4. 核心代码讲解（答辩重点）

### 4.1 HDFS 操作 — `behavior/service/HdfsService.java`
- `FileSystem.get(URI, conf, user)`：用配置的 `fs.defaultFS` 和用户名连 NameNode。
- 写：`fs.create(path)` 拿输出流，按「天」分目录、一批一个文件，避免 HDFS 小文件 + 并发 append 问题。
- 读：遍历目录下 `part-r-*`，`fs.open()` 逐行读回，交给分析层解析。
- **NAT 关键点**：`dfs.client.use.datanode.hostname=true` —— 服务器对外是公网
  `218.194.47.8`，但 DataNode 注册的是内网主机名 `master`；开启此项 + 客户端
  hosts 映射 `218.194.47.8 master`，客户端才能正确连到 DataNode 读写数据块。

### 4.2 采集不侵入业务 — `BehaviorInterceptor`
Spring MVC 拦截器，只看 URL+方法识别行为（`GET /posts/{id}`→VIEW 等），
不改任何已有 Controller。失败请求(非2xx)不记录。埋点失败只记日志、绝不阻塞用户。

### 4.3 MapReduce — `geng-mr`
- **HotPostJob**：Mapper 把每行行为按权重折算（收藏5/评论3/发帖2/浏览1）输出
  `(postId, weight)`；Reducer 求和得帖子热度；单 Reducer + `TreeMap` 在
  `cleanup()` 里取全局 Top-N，保证排序全局正确。
- **ActionDistJob**：标准 WordCount 变体，`(action,1)` → 求和；加 `Combiner`
  减少 shuffle 数据量。
- jar 里**只有我们 3 个类**，Hadoop 类由 `hadoop jar` 命令注入（依赖 `compileOnly`）。

### 4.4 大数据 + 业务融合点
`AnalyticsService` 把 **MapReduce 算出的热度**（HDFS）和 **帖子标题**（MySQL）
join 起来返回——这一步最能体现"两套系统融合"。

## 5. 环境现状与已知问题（答辩要主动讲，体现你懂原理）

通过 SSH 实测，服务器当前状态：

| 项 | 现状 | 说明 |
|---|---|---|
| Hadoop / Java | 3.4.1 / Java 21 | 与客户端版本一致，无兼容问题 |
| NameNode RPC | `hdfs://218.194.47.8:10035` | 实测确认（非默认 9000） |
| **DataNode** | **未运行，HDFS 安全模式** | clusterID 不匹配（NameNode 被 format 过），DataNode 注册被拒退出 |
| 网络 | NAT，内网 `192.168.31.219`=`master` | 需 hosts 映射 + use-datanode-hostname |
| YARN 内存 | NodeManager 1GB，机器 2GB | `mapred-site.xml` 已调小，作业别再加大内存 |

**DataNode 修复思路（环境就绪后执行，本项目代码无需改动）**：
停 HDFS → 清理 DataNode 数据目录 `/opt/hadoop/tmp/dfs/data`（或同步 VERSION 的
clusterID）→ 重启 DataNode → `hdfs dfsadmin -safemode leave`。

## 7. 可能被问到的问题

- **为什么不用 Spring Data Hadoop / HadoopTemplate？** 该项目 2019 年起停更，
  不兼容 Spring Boot 3/4 与 Jakarta EE，所以用官方 Hadoop Java API（更底层、更可控）。
- **为什么 MapReduce 不从 Spring Boot 提交而是打 jar 在服务器跑？** Windows 客户端
  跨 NAT 向远程 YARN 提交作业有 winutils/跨平台/staging 权限等一堆坑；打 jar 用
  `hadoop jar` 跑是教科书做法，稳定且职责清晰（SB 只负责 HDFS 读写）。
- **Top-N 为什么要单 Reducer？** 多 Reducer 各自只能算局部 Top-N，合并后不是全局
  最优；单 Reducer + TreeMap 才能保证全局正确（数据量小，单 Reducer 没有性能问题）。
- **埋点会拖慢系统吗？** 不会。拦截器只入内存队列（无锁、O(1)），HDFS 写在
  独立定时任务里批量做，且任何 HDFS 异常都被吞掉只记日志。
- **Java 21 跑 Hadoop 为什么要加 `--add-opens`？** Hadoop 用了 JDK 内部 API 反射，
  Java 17+ 模块系统默认禁止，需显式开放（已写进 `build.gradle` 的 bootRun）。
